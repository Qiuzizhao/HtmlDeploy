# AI 命名审核任务防卡死设计

## 背景

生产环境的全部命名审核任务 `00b8562d` 停在 210/492。任务状态仍保留在进程内存中并标记为 `running`，但执行任务的异步 worker 已经消失，最后一次心跳停在 2026-07-18 13:18:46。最新的任务重连逻辑会持续找回这个僵尸状态，导致后台永久显示正在审核。

本次修复只解决运行中任务的防卡死和僵尸状态恢复。任务跨服务重启的持久化断点续跑不在本次范围内。

## 目标

- 任意单个项目的审核流程都不能让整批任务永久停住。
- 单项超时后将该项计为失败，并继续处理后续项目。
- 已超时的异步操作即使稍后返回，也不能再修改项目名称。
- worker 自身发生意外异常时，任务必须进入可见的 `error` 状态。
- 管理页面不能重新连接已经长时间没有心跳的僵尸任务。
- 当前任务被 PM2 重启清除后，已经完成的 210 项数据保持不变，管理员可以重新发起整批审核。

## 非目标

- 不将批量任务或项目快照持久化到 SQLite。
- 不实现服务重启后的断点续跑。
- 不并行调用 AI；仍保持串行审核，避免请求突发和并发写入。
- 不更改单项 AI 命名审核的判断规则、提示词或违禁词行为。

## 方案比较

### 方案 A：进程内 watchdog 与僵尸识别（采用）

为每个项目的完整审核流程设置硬超时，并使用尝试令牌阻止过期操作写回；同时保存 worker Promise、维护心跳并拒绝恢复过期任务。改动集中、无需数据库迁移，直接覆盖本次故障模式。

### 方案 B：SQLite 持久化断点续跑

把任务、项目游标、事件和状态写入数据库，服务启动时恢复任务。它能覆盖进程重启，但需要任务状态机、迁移、幂等恢复和重复写入防护，超出这次修复的必要范围。

## 后端设计

### 单项完整流程超时

`runAiNameReviewJob` 为每个 `siteSnapshot` 创建独立尝试令牌，并通过通用的超时包装器执行 `reviewAndMaybeRenameSite`。默认硬超时应高于现有 AI 请求默认超时 45 秒，采用 60 秒；测试可通过 `createApp` 选项注入更短时间。

超时发生时：

1. 令牌立即失效。
2. 当前项目增加一次 `failed`。
3. 添加明确的超时错误事件。
4. `finished` 增加一次。
5. 循环继续处理下一项目。

超时包装器必须自行结束等待，不能依赖底层 `fetch` 的 AbortController 是否正确回调。

### 过期操作写入保护

`reviewAndMaybeRenameSite` 接受可选的活动检查回调。在任何项目名称写入之前检查该回调；尝试已超时或已被替代时抛出过期错误，不执行 `writeSites`。只读结果可以完成，但不能产生迟到写入。

### worker 生命周期

创建任务时保存 worker Promise，并为 Promise 安装最终错误处理。任何逃出 `runAiNameReviewJob` 正常状态处理的异常都必须：

- 把仍为 `queued` 或 `running` 的任务更新为 `error`；
- 清空 `current`；
- 写入 `finishedAt` 和错误事件；
- 释放 `activeAiNameReviewJobId`；
- 清除 API Key 引用和 worker 引用。

任务对象的公开响应不暴露 worker、令牌或 AI 配置。

### 心跳与僵尸任务

任务的 `updatedAt` 继续作为心跳，每次开始项目、完成项目、记录事件或更新状态时刷新。

在创建新任务、查询当前任务和查询指定任务时进行僵尸检查。对 `queued` 或 `running` 且超过 90 秒没有心跳的任务：

- 标记为 `error`；
- 清空 `current`；
- 设置 `finishedAt`；
- 添加“任务长时间无进展，已自动结束”的错误事件；
- 释放活动任务 ID；
- 不再由 `/api/admin/ai-name-review-jobs/current` 返回为当前运行任务。

90 秒高于 60 秒单项硬超时，为正常清理和事件写入保留余量。

## 前端设计

轮询指定任务时如果收到 `error` 状态，沿用现有失败分支，但提示语明确为“全部命名审核任务已结束，请查看日志后重新发起”。本地保存的 jobId 随后清除，按钮恢复可用。

页面启动时调用 `/current` 得到 404，表示当前没有可恢复的有效任务；不显示错误。这样过期任务不会再次附着到按钮。

## 测试设计

新增自动化回归测试覆盖：

1. 第一项目的审核 Promise 永远不返回时，单项硬超时将其计为失败，第二项目仍被处理，任务最终完成。
2. 已超时的第一项目后来返回无关结论时，不会迟到改名。
3. `updatedAt` 超过僵尸阈值的运行任务不会被 `/current` 恢复，并进入 `error`。
4. worker Promise 意外拒绝时，任务不会永久保持 `running`。
5. 现有批量串行、重复创建返回同一活动任务、事件游标和正常完成测试继续通过。

测试使用注入的毫秒级超时和本地假 LLM 服务，不访问生产 AI。

## 部署与恢复

完成完整测试后提交并推送 `main`。生产服务器执行快进拉取和 PM2 重启；重启会清除当前僵尸内存任务，但不会回滚已经写入 SQLite 的前 210 项结果。

部署验证包括：

- 生产 HEAD 与远端 `main` 一致；
- PM2 中 `html-deploy` 为 `online`；
- 本机端口 3005 和公网首页返回 HTTP 200；
- `/api/admin/ai-name-review-jobs/current` 不再返回旧任务 `00b8562d`；
- 管理页面仍包含全部命名审核控件与恢复逻辑。
