# 网站流畅度体验优化方案

更新时间：2026-06-08

## 一、背景与目标

当前项目站已经从最初的 demo 逐步增加了班级密码、项目前台展示、后台项目管理、封面生成、AI 优化、违禁词、查重、星标、使用次数、GitHub 同步等功能。功能变多以后，页面偶尔出现“点一下要等一会儿”“切换班级或打开后台时卡一下”“后台操作后整页重新加载”的体验问题，是正常会暴露出来的性能瓶颈。

本方案目标不是把系统改复杂，而是在保持现有 Express + JSON 文件 + 单 HTML 前后台结构的前提下，优先解决最影响使用流畅度的问题：

1. 前台首页切换班级、搜索、打开项目更快。
2. 后台项目管理页加载更快，几百个项目也不卡。
3. 封面生成、AI 优化、GitHub 同步这类重任务不阻塞页面。
4. 页面有明确的加载状态、局部刷新和失败提示。
5. 为后续项目数量继续增长预留简单升级路线。

## 二、当前技术现状

目前技术栈：

- 后端：Node.js + Express。
- 数据：`data/sites.json`、`data/classes.json`、`data/settings.json` 等 JSON 文件。
- 前台页面：`public/index.html`，单文件包含 HTML、CSS、JS。
- 后台页面：`public/admin.html`，单文件包含 HTML、CSS、JS。
- 项目文件：保存在 `storage/sites/<项目ID>/index.html`。
- 封面图：保存在 `storage/thumbnails/<项目ID>.png`。
- 封面生成：后端用 Playwright 打开项目页面并截图。
- 自动同步：数据写入后触发 Git add/commit/push。

当前文件体量大致为：

- `public/index.html`：约 55KB。
- `public/admin.html`：约 131KB。
- `src/app.js`：约 94KB。
- `data/sites.json`：约 76KB，已有较多项目数据。

这说明目前的主要问题不是“技术栈太重”，而是“很多操作都一次性做完、一次性渲染、一次性计算”，项目数量上来后自然会卡。

## 三、主要卡顿来源判断

### 1. 前台一次性加载和渲染项目卡片

前台 `/api/sites` 会返回当前班级或全部项目，然后 `index.html` 一次性把项目卡片全部创建出来。每个卡片包含封面、标题、编号、班级、作者、次数、链接、按钮。项目数量越多，DOM 创建越多，浏览器主线程就越容易短暂停顿。

影响场景：

- 打开首页。
- 输入全部密码后展示所有项目。
- 切换到项目很多的班级。
- 搜索或切换星标筛选。

### 2. 后台项目列表一次性加载所有项目并计算容量

后台 `/api/admin/sites` 当前会读取所有项目，并且对每个项目调用目录大小统计。这个统计需要访问文件系统，如果项目多，每次进入后台项目管理页都要遍历一遍 `storage/sites`，会造成明显等待。

影响场景：

- 打开后台项目管理。
- 每次新增、编辑、删除、星标、启用禁用后重新加载列表。
- 点击“刷新”后台数据。

### 3. 后台一次性渲染大表格

后台 `admin.html` 会把所有项目渲染成表格行，每行包含输入框、下拉框、封面、很多操作按钮。几百行一起创建 DOM，会比前台卡片更容易卡。

影响场景：

- 后台项目管理初次加载。
- 操作某一个项目后调用 `loadSites()`，导致整张表重渲染。
- 排行榜、汇总统计、表格同时刷新。

### 4. 使用次数统计会频繁写 JSON 并触发 Git 同步

用户打开项目预览或查看代码时，会更新使用次数并写入 `sites.json`。写 JSON 本身不算重，但每次写入都会触发 GitHub 自动同步排队。访问量稍微高一点时，写入和同步会形成持续后台压力。

影响场景：

- 很多学生同时点开项目。
- 前台卡片使用次数频繁变化。
- 服务器后台持续出现 Git 同步任务。

### 5. 封面生成属于重任务

封面生成需要启动 Chromium、打开页面、等待、截图。单个项目还可以接受，批量生成全部封面时会非常重，而且失败时用户只能看到最终失败数量，不知道进度和原因。

影响场景：

- 上传项目后后台自动生成封面。
- 后台点击“刷新全部封面”。
- AI 优化保存后重新生成封面。

### 6. 后台启动时串行请求较多

后台初始化会依次加载设置、AI 设置、违禁词、班级、项目。只要其中一个接口慢，整个后台就显得慢。尤其项目列表接口最重，会拖慢整体进入体验。

### 7. 前后台单 HTML 文件越来越大

现在前后台都把结构、样式、脚本写在一个 HTML 文件里，优点是简单，缺点是：

- 浏览器每次加载 HTML 都要解析较大的脚本和样式。
- 很难只缓存 JS 或 CSS。
- 后续改动越来越容易影响整页。

短期可以不拆，但中期建议拆成 `index.css`、`index.js`、`admin.css`、`admin.js`。

## 四、优化原则

1. 先优化感知速度，再优化内部架构。
   用户先看到页面、按钮先能点、后台先显示骨架，比“所有数据一次性完整加载”更重要。

2. 先做缓存和分页，少做大重构。
   现有系统还不需要马上换数据库，先把 JSON 读写、目录容量、列表渲染变轻。

3. 重任务全部后台化。
   封面生成、AI 优化、Git 同步、查重、违禁词审查都应该是任务状态，而不是让用户一直等一个请求。

4. 后台操作尽量局部更新。
   星标、启用、保存标题、改班级后，只更新当前行和汇总，不要每次整张表重新渲染。

5. 保留简单部署方式。
   优化方案优先兼容当前 Ubuntu + PM2 + Nginx + GitHub 同步流程。

## 五、第一阶段：快速止血优化

第一阶段目标：改动不大，但明显减少卡顿。建议优先完成。

### 1. 后台项目列表增加分页

新增接口参数：

```text
GET /api/admin/sites?page=1&pageSize=50&classId=&q=&starred=&enabled=
```

返回结构：

```json
{
  "items": [],
  "total": 268,
  "page": 1,
  "pageSize": 50,
  "summary": {
    "totalProjects": 268,
    "enabledProjects": 250,
    "disabledProjects": 18,
    "totalStorageBytes": 12345678
  }
}
```

后台页面默认每页 50 个项目，并提供：

- 上一页 / 下一页。
- 每页 20 / 50 / 100。
- 按班级筛选。
- 搜索项目名称、作者、编号。
- 只看禁用、只看星标。

预期效果：

- 后台不再一次性渲染几百行。
- 打开后台项目管理明显变快。
- 操作按钮响应更轻。

### 2. 后台项目容量改为缓存

当前每次 `/api/admin/sites` 都实时遍历项目目录计算容量。建议新增容量缓存字段：

```json
{
  "storageBytes": 12345,
  "storageUpdatedAt": "2026-06-08T09:00:00.000Z"
}
```

更新时机：

- 上传项目后计算一次。
- 修改项目代码后重新计算一次。
- 删除项目后从总容量扣除。
- 后台提供“重新统计容量”按钮，用于手动修正。

接口加载项目列表时直接读取缓存，不再遍历目录。

预期效果：

- `/api/admin/sites` 从“文件系统遍历型接口”变成“读 JSON 接口”。
- 后台项目列表加载时间会明显下降。

### 3. 前台列表增加分页或“加载更多”

前台不一定要传统分页，可以更适合学生使用：

- 默认显示前 24 个项目。
- 页面底部显示“加载更多”按钮。
- 搜索时只在当前已加载数据或请求后端搜索。
- 班级项目很多时，也不会一次性创建所有卡片。

推荐接口：

```text
GET /api/sites?classId={classId}&page=1&pageSize=24&q=&starred=
```

返回：

```json
{
  "items": [],
  "total": 80,
  "page": 1,
  "pageSize": 24
}
```

预期效果：

- 首页初次展示更快。
- 切换班级更顺。
- 手机或性能一般的电脑上不会因为大量卡片短暂停顿。

### 4. 前台封面图片懒加载和固定尺寸

已有 `loading="lazy"`，建议继续补充：

- 封面图片增加固定宽高，减少布局跳动。
- 缩略图返回 WebP 或压缩 PNG。
- 后端封面生成尺寸控制在卡片实际展示需要的大小，例如 480x270。
- Nginx 给 `/thumbnails` 添加长期缓存。

建议响应头：

```text
Cache-Control: public, max-age=31536000, immutable
```

因为缩略图 URL 已经带 `?v=mtime`，图片变化时链接会变，不需要担心缓存过期。

### 5. 后台初始化改为并行加载

后台启动时可以并行加载轻接口：

```js
await Promise.all([
  loadSettings(),
  loadAiSettings(),
  loadForbiddenWords({ offset: 0 }),
  loadClasses()
]);
await loadSites();
```

更进一步：

- 先显示后台布局和菜单。
- 项目管理页被打开时再加载项目列表。
- 设置页被打开时再加载 AI 设置和违禁词。

预期效果：

- 后台首屏更快出现。
- 用户不进入的页面不请求数据。

### 6. 按钮操作改为局部更新

这些操作不应该每次都整表 `loadSites()`：

- 星标。
- 启用 / 禁用。
- 加入 / 移出白名单。
- 保存标题、作者、班级。
- 单个封面生成。

接口返回更新后的单个项目后，前端只替换 `sites[index]` 并更新当前行。

预期效果：

- 后台点击按钮后几乎马上反馈。
- 减少重复请求 `/api/admin/sites`。

## 六、第二阶段：任务队列优化

第二阶段目标：让重任务不再阻塞页面，并且用户能看到进度。

### 1. 封面生成改成任务队列

新增任务接口：

```text
POST /api/admin/thumbnails/jobs
GET /api/admin/thumbnails/jobs/:jobId
```

任务状态示例：

```json
{
  "id": "{jobId}",
  "status": "running",
  "total": 268,
  "finished": 35,
  "success": 30,
  "failed": 5,
  "current": "00268 双人连招格斗",
  "errors": [
    { "siteId": "abc", "title": "项目名", "message": "打开页面超时" }
  ]
}
```

后台体验：

- 点击“刷新全部封面”后立即返回任务 ID。
- 页面显示进度条、成功数量、失败数量。
- 用户切换页面或刷新后，仍能看到任务状态。
- 失败项目可单独重试。

### 2. 限制 Playwright 并发

封面生成建议并发为 1 或 2，不要同时启动很多 Chromium。

推荐：

- 单个封面：进入队列，但优先级高。
- 全部封面：低优先级批量任务。
- 上传后自动封面：延迟 3 到 10 秒生成，避免上传高峰时立刻压垮服务器。

### 3. AI 优化日志持久化

已有 AI 优化日志需求，建议统一成任务日志模型：

```json
{
  "jobId": "{jobId}",
  "siteId": "{siteId}",
  "type": "ai-optimize",
  "status": "success",
  "logs": [],
  "createdAt": "",
  "updatedAt": ""
}
```

日志保存在 `data/jobs.json` 或 `data/ai-jobs.json`，而不是只放内存 Map。

好处：

- 刷新后日志还在。
- 后台可以展示最近任务。
- 出问题时更容易定位。

### 4. GitHub 同步降频

当前每次写入数据都会触发同步排队。建议：

- 使用次数统计不立即触发 GitHub 同步。
- 普通数据修改延迟 30 到 60 秒合并同步。
- 大量任务结束后只同步一次。
- 后台显示“上次同步时间”和“待同步变更数”。
- 保留手动“立即同步到 GitHub”按钮。

推荐策略：

| 数据类型 | 是否立即同步 | 建议 |
| --- | --- | --- |
| 新增项目 | 是，延迟几秒合并 | 保证作品不丢 |
| 删除项目 | 是，延迟几秒合并 | 保证数据一致 |
| 班级设置 | 是 | 重要配置 |
| 违禁词 | 是 | 管理配置 |
| 使用次数 | 否，定时批量 | 高频低风险 |
| AI 日志 | 否，定时批量 | 可批量 |
| 缩略图文件 | 可选 | 不一定必须进 Git |

## 七、第三阶段：数据层优化

第三阶段目标：项目继续增长到上千个时仍然顺畅。

### 1. JSON 读写增加内存缓存

当前很多接口都会 `readSites()`、`readClasses()`、`readSettings()`。可以做简单内存缓存：

- 启动时读入内存。
- 写入时更新内存并落盘。
- 使用写队列避免并发写覆盖。
- 文件被外部更新时，按 mtime 检测重新加载。

好处：

- 接口不再反复读 JSON 文件。
- 高频访问 `/api/sites` 更轻。

### 2. 使用次数单独存储

使用次数是高频变更，不适合和项目基础信息放在同一个 `sites.json` 里。建议拆成：

```text
data/sites.json        项目基础信息
data/site-usage.json   使用次数统计
```

`site-usage.json` 示例：

```json
{
  "742b8763": {
    "preview": 12,
    "code": 3,
    "lastUsedAt": "2026-06-08T09:00:00.000Z"
  }
}
```

更进一步可以在内存中累加，每 30 秒落盘一次。

### 3. 项目列表维护派生索引

可以在内存中维护这些索引：

- 按班级分组。
- 按创建时间排序。
- 按编号排序。
- 星标项目集合。
- 禁用项目集合。

这样 `/api/sites?classId={classId}` 不需要每次遍历所有项目。

### 4. 中长期可考虑 SQLite

如果项目数超过 1000，或者访问人数明显增加，建议把 JSON 数据迁移到 SQLite。

SQLite 优点：

- 单文件，部署仍然简单。
- 支持分页、搜索、排序。
- 写入更安全。
- 不需要额外数据库服务。

建议迁移表：

- `sites`
- `classes`
- `settings`
- `site_usage`
- `jobs`
- `forbidden_words`

不建议一开始就上 MySQL/PostgreSQL，当前项目规模用 SQLite 更合适。

## 八、前台体验优化细节

### 1. 班级切换体验

切换班级时：

- 立即高亮按钮。
- 项目区域显示骨架屏，不要空白。
- 如果需要密码，显示密码输入框。
- 如果接口慢，显示“正在加载项目...”。
- 加载完成后局部替换卡片。

### 2. 搜索体验

当前搜索可以继续保留前端搜索，但项目多以后建议：

- 小于 100 个项目：前端搜索。
- 大于 100 个项目：后端搜索。
- 输入时使用 300ms 防抖，避免每打一个字都重渲染。

### 3. 项目卡片优化

建议：

- 卡片固定高度或固定封面比例。
- 图片未加载前显示轻量占位。
- 按钮点击后只更新按钮状态。
- 卡片渲染分批进行，例如每帧渲染 12 个，避免一次性阻塞。

### 4. 项目打开体验

点击“预览项目”后：

- 立即打开大窗口或 iframe。
- 同时后台异步增加使用次数。
- 使用次数更新失败不影响用户打开项目。

也就是说，统计不能阻塞项目访问。

## 九、后台体验优化细节

### 1. 后台首页不要默认加载所有重数据

后台进入后可以先显示：

- 总项目数。
- 班级数。
- 待处理任务。
- 最近上传项目。

项目管理表格在点击“项目管理”后再加载。

### 2. 项目管理表格虚拟滚动或分页

优先推荐分页，因为实现简单、稳定。

如果以后想体验更像本地软件，可以做虚拟滚动：

- 页面只渲染可见的 30 到 60 行。
- 滚动时替换 DOM。
- 表头固定。

但虚拟滚动实现比分页复杂，第一轮不建议优先做。

### 3. 操作列精简

当前操作列按钮较多，建议分组：

- 常用：保存、预览、代码、星标、启用。
- 更多：AI 命名、AI 优化、下载、生成封面、删除。

可以做一个“更多”下拉菜单，减少每行 DOM 节点数量。

### 4. 表格行局部状态

每一行应该有自己的状态：

- 保存中。
- 已保存。
- 失败。
- 生成封面中。
- AI 优化中。

不要只依赖顶部 `listMessage`，否则用户不知道是哪一行在处理。

### 5. 违禁词管理继续分页

违禁词已经做了分页搜索，这是正确方向。后续可以加：

- 添加后只刷新当前搜索结果。
- 删除后只移除当前词条。
- 大批量导入时显示进度。

## 十、后端接口优化建议

### 1. 增加接口耗时日志

先给关键接口加日志，定位真实慢点：

```text
[perf] GET /api/admin/sites 386ms sites=268 storage=true
[perf] GET /api/sites 45ms classId=c37011bf count=80
[perf] POST /api/admin/thumbnails 120000ms total=268 failed=45
```

重点记录：

- 请求路径。
- 耗时。
- 返回数量。
- 是否计算容量。
- 是否触发 Git 同步。

### 2. 增加轻量健康接口

```text
GET /api/health
```

返回：

```json
{
  "ok": true,
  "uptime": 12345,
  "version": "0.1.0",
  "sites": 268,
  "classes": 10,
  "pendingGitSync": false,
  "runningJobs": 1
}
```

便于判断服务器是真卡，还是浏览器端卡。

### 3. 静态资源缓存

Nginx 建议：

- `/thumbnails` 长缓存。
- `/site/<id>` 可短缓存或不缓存，避免学生作品修改后不更新。
- `/public` 静态资源拆分后可长缓存并带 hash。

### 4. 压缩响应

可以在 Express 增加 `compression` 中间件，或者让 Nginx 开启 gzip。

适合压缩：

- HTML。
- CSS。
- JS。
- JSON。

不需要压缩：

- PNG。
- 已压缩图片。

## 十一、推荐实施顺序

### 第 1 轮：最小改动提速

目标：一两天内能看到明显改善。

1. 后台 `/api/admin/sites` 增加分页参数。
2. 后台项目表默认每页 50 条。
3. 项目容量改为缓存字段，不再每次列表请求实时遍历目录。
4. 后台星标、启用、白名单、保存改为局部更新。
5. 后台启动改为按页面懒加载数据。
6. 前台搜索增加防抖，卡片分批渲染。

预期收益：

- 后台打开速度明显提升。
- 项目管理操作不再频繁整页卡顿。
- 前台班级切换更顺。

### 第 2 轮：重任务后台化

目标：封面、AI、同步不再拖慢页面。

1. 封面生成改成任务队列。
2. 批量封面显示进度和失败明细。
3. AI 优化日志持久化。
4. GitHub 同步降频，使用次数不立即同步。
5. 后台增加任务中心或任务状态区域。

预期收益：

- “刷新全部封面”不再让页面长时间等待。
- 刷新后台后任务状态还在。
- Git 同步不再频繁抢资源。

### 第 3 轮：数据结构升级

目标：支持更多项目和更高访问量。

1. `sites.json` 增加内存缓存和写队列。
2. 使用次数拆成 `site-usage.json`。
3. 项目列表增加内存索引。
4. 如项目数继续增长，迁移 SQLite。

预期收益：

- 高频访问更稳。
- 写入冲突更少。
- 后续功能扩展更容易。

## 十二、验收指标

建议在优化前后记录以下指标：

| 场景 | 当前可能问题 | 目标 |
| --- | --- | --- |
| 打开前台首页 | 首屏等待、卡一下 | 1 秒内显示班级和基本布局 |
| 切换班级 | 项目多时卡顿 | 500ms 内出现加载状态，2 秒内完成 |
| 打开后台 | 等所有数据加载 | 1 秒内出现后台框架 |
| 后台项目列表 | 几百行一次性渲染 | 每页 50 条，1 秒内显示 |
| 星标/禁用项目 | 操作后整表刷新 | 当前行 300ms 内反馈 |
| 生成全部封面 | 长时间无进度 | 立即返回任务，持续显示进度 |
| 使用次数统计 | 高频写入拖慢 | 不影响项目打开 |
| GitHub 同步 | 频繁触发 | 合并批量同步 |

## 十三、测试与验证清单

每轮优化完成后需要验证：

1. 前台能正常显示班级按钮。
2. 选择班级后密码逻辑正常。
3. 刷新后保留已选班级。
4. 前台项目卡片显示编号、班级、星标、次数。
5. 搜索和星标筛选正常。
6. 点击预览项目能打开大窗口。
7. 点击查看代码能显示代码。
8. 后台登录正常。
9. 后台项目分页、搜索、筛选正常。
10. 新增、编辑、删除、启用、禁用、星标项目正常。
11. 班级新增、编辑、删除、密码设置正常。
12. 违禁词分页、搜索、添加、删除正常。
13. 查重和违禁词审查不会阻塞页面。
14. AI 优化任务日志刷新后仍存在。
15. 封面生成任务有进度，失败项目可追踪。
16. GitHub 手动同步正常。
17. 自动同步不会频繁重复提交。
18. 服务器 PM2 在线，首页 HTTP 200。

## 十四、建议优先做的具体任务

如果马上开始优化，推荐从这 8 个任务开始：

1. 给后台项目列表做分页。
2. 给项目容量做缓存字段。
3. 后台表格操作改为局部更新。
4. 前台卡片分批渲染和搜索防抖。
5. 后台数据按页面懒加载。
6. 封面生成改任务队列。
7. 使用次数从即时 Git 同步中拆出来。
8. 增加接口耗时日志。

这 8 个任务做完，整个网站的“经常卡一下”会明显减少，而且不会把系统改得太复杂。

## 十五、最终推荐路线

短期不要急着换技术栈，也不需要立刻上数据库。当前最合适的路线是：

1. 保持 Express + JSON 文件。
2. 先加分页、缓存、局部更新和任务队列。
3. 等项目数或访问量继续上涨，再把数据迁移到 SQLite。

这样既能快速改善体验，又不会破坏现在已经能运行、能部署、能同步 GitHub 的简单结构。
